# CHANGELOG

## 2.10
- **«Хронология загрузки» переработана в раскрываемые фазы.** Каждая веха — фаза с
  её длительностью (дельта от предыдущей) и Gantt-полосой (где во времени и сколько
  заняла); самая долгая фаза подсвечивается. По клику фаза раскрывается и показывает
  реальные строки `dmesg`, выполнявшиеся в ней — видно, что именно съедает время
  (обычно «хвост» после procd — инициализация модема/USB/Wi-Fi). Добавлена
  синтетическая финальная веха «Система готова» и аптайм в блоке System. `boot.awk`
  теперь отдаёт массив `log`; таймлайн строится **один раз** (раскрытия не слетают
  при поллинге), поллится только блок System.

## 2.9
- **Починена «Хронология загрузки» по-настоящему** (снимок dmesg в v2.5 был верным,
  но парсер `boot.awk` не работал на роутере). Причина: `gsub(/[\[\] ]/,…)` —
  класс с экранированными скобками внутри `[...]` — gawk/mawk понимают, а **BusyBox
  awk нет**, поэтому таймстамп оставался `[12.3]`, `+0` давал `0`, и `available`
  становился `false` («недоступно»). Заменено на переносимое `gsub(/[^0-9.]/,…)`.
  Подтверждено на устройстве: загрузка парсится как ~29.7 с. (Юнит-тест не ловил
  баг, т.к. Git Bash использует другой awk.)

## 2.4
- **Убран `net.core.netdev_budget_usecs`** из рекомендаций. На ImmortalWRT 6.12
  ядро держит его на **20000 µs** и **отвергает запись 4000** (`sysctl: write
  error: Invalid argument` — есть минимум выше 4000). А 20000 µs и так с большим
  запасом перекрывают цель (чтобы 600-пакетный бюджет NAPI отрабатывал раньше, чем
  истечёт таймер). Значение 4000 было взято из устаревшего предположения о дефолте
  2000 и являлось бы понижением — параметр исключён. `netdev_budget`=600 и
  `netdev_max_backlog`=5000 остаются.
- **Тумблер параметра перенесён вправо** (после колонки «Статус»).
- **Показаны ВСЕ изменяемые IPv6-параметры** (раньше было только 2 sysctl, а по
  факту меняется больше): добавлены `network.wan.ipv6` (IPv6 на WAN), `dhcp.dhcpv6`/
  `dhcp.ra`/`dhcp.ndp` (выдача по всем DHCP-пулам), `network.globals.ula_prefix`
  (удаление ULA), `service.odhcpd` (остановка демона) — каждый со своим статусом и
  тумблером. Откат **state-driven и точный**: перед каждым изменением исходное
  значение каждого uci-опшена сохраняется в state (`uci:<path>` → значение/`__unset__`),
  «Сбросить всё» восстанавливает **именно их** (а не дефолты) и трогает только то,
  что реально меняли — чужие настройки пулов/делегирования не затрагиваются.
- **Умный статус IRQ-балансинга по числу ядер.** Рекомендация теперь зависит от CPU:
  **<4 ядер → OFF** (показывается «stopped / not installed»), **≥4 ядер → running**.
  Если рекомендуется OFF, то текущее `stopped` **или** `not installed` даёт статус
  **Matches** (не установлен = не работает = цель достигнута). Если рекомендуется
  running, а пакет не установлен — **Unavailable**.

## 2.3
- **Переработана модель статусов.** Убран статус «По умолчанию» (`@default`).
  Теперь ровно **3 статуса**: **Applied** (значение изменили мы — факт
  запоминается), **Matches** (совпало с рекомендованным само), **Off**
  (отличается, ещё не приведено). Шкала соответствия = **(Applied + Matches) /
  всего** (Unavailable не учитывается).
- **Индивидуальный тумблер у КАЖДОГО параметра** + мастер-переключатель в шапке
  блока (вкл/выкл все параметры блока разом, с промежуточным состоянием). По
  умолчанию включены ВСЕ рекомендации; тумблер лишь исключает параметр из
  следующего «Применить» — откат значений только через «Сбросить всё».
- **Конкретные значения вместо `@default`** для 11 параметров: `rmem_default`/
  `wmem_default`=262144, `optmem_max`=65536, `tcp_rmem`/`tcp_wmem`=
  «4096 131072 4194304», `tcp_tw_reuse`=1, `tcp_fin_timeout`=15,
  `tcp_max_syn_backlog`=8192, `tcp_max_tw_buckets`=**65536** (а не 200000 — 50 МБ
  RAM на 512 МБ роутере избыточно), `netdev_max_backlog`=5000, `netdev_budget`=600.
  Добавлен **`netdev_budget_usecs`=4000** — иначе временной лимит NAPI срабатывает
  раньше пакетного, и подъём `netdev_budget` нивелируется.
- **flow offload и irqbalance: рекомендация = ВЫКЛ** (`rec` 0 / stopped). Когда уже
  выключены — статус **Matches**; включённый тумблер «Применить» приводит их к off
  (offload несовместим с fq_codel AQM и policy-routing Podkop; irqbalance на 2 ядрах
  нейтрально/вредно).
- **Память применённого + точный откат.** Исходные значения изменённых параметров
  пишутся в `/etc/streamtune.applied` (key⇥orig). Это отличает Applied от Matches и
  позволяет «Сбросить всё» восстановить именно исходные значения, затем чистит память.
- **Хранение тумблеров:** выключенные параметры — в UCI-списке
  `streamtune.global.param_off` (пусто = все включены); устаревшие per-category
  опции вычищаются на обновлении (uci-defaults).
- **softnet_stat в Diagnostics** — суммарные RX-дропы (col2) и NAPI-squeeze (col3)
  по CPU; справочно показывает, насыщается ли очередь/бюджет (польза `netdev_*`).
- **Проба MTU**: ищет DF-способный `ping` по полному пути (`/usr/bin/ping` →
  `/bin/ping`, тест на loopback) — BusyBox `ping` не умеет `-M do`, и голый вызов
  упирался в него даже при установленном `iputils-ping`.

## 2.2
- Кнопка **«Определить оптимальный MTU»** теперь **подставляет найденное значение
  в колонку «Рекомендовано»** строки MTU и сохраняет его (UCI `mtu`) при
  «Применить выбранное» — раньше результат показывался лишь сбоку, а в реестре
  оставался `auto`.
- MTU стал переопределяемым: `streamtune.global.mtu` = `auto` (проба) или число
  (576..9000); `apply` пишет его, `detect` показывает в «Рекомендовано».
- **Починка carrier-MTU из `mmcli`:** ключ bearer имеет формат
  `modem.generic.bearers.value[1]` (квадратные скобки), а парсер искал `value.1`
  (точка) → bearer не находился, carrier-MTU не читался. Добавлен
  `st_modem_bearer()` (берёт DBus-путь bearer независимо от формата индекса),
  `st_modem_iface`/`st_modem_mtu` используют его.
- **Проба MTU переписана** (`mtu_probe.sh`): приоритет — **carrier-MTU оператора**
  (через `mmcli`, авторитетно для сотового и не требует ICMP) + **DF-ping**
  (бинарный поиск, устойчив к фильтрации ICMP), `tracepath -m 12` как запасной;
  итог = `min(measured, carrier)`. Прежний код звал `tracepath` с чужими опциями
  `-I`/`-m <MTU>` (это опции `ping`; у `tracepath` `-m` — это max-hops 0..255) →
  «invalid argument: out of range» и фолбэк на MTU интерфейса.
- Зависимость **`iputils-tracepath` → `iputils-ping`**: BusyBox `ping` не умеет
  `-M do`, а `iputils-ping` надёжнее `tracepath` при фильтрации ICMP на сотовых.

  анализа (см. [`tuning-lte-audio.md`](tuning-lte-audio.md)).
- **Профили: «Авто LTE» + «Дом, провод»** («Базовый» убран). Значения выверены:
  буферы 4 МБ; серверные TIME_WAIT/backlog-крутилки и 16 МБ `*_default` →
  дефолт ядра (не управляем); `udp_*_min`=8192.
- **Новая категория «Mobile LTE link»**:
  - **MTU auto-probe** — DF-проба/`tracepath` + carrier-MTU из `mmcli`, берётся min;
  - **MSS-clamp** через явное **pbr-safe nftables-правило** (`oifname`+`iifname` WAN,
    `tcp option maxseg size set rt mtu`, обе стороны) вместо зонного `mtu_fix` —
    корректно при Podkop/policy-routing;
  - **conntrack established timeout = 7440**.
- `default_qdisc` → **fq_codel** (не fq); **fq_codel больше не завязан на BBR** (фикс:
  пропуск bbr по ключу, а не по категории).
- **Полное отключение IPv6**: стек + WAN `ipv6=0` + все DHCP-пулы
  (`dhcpv6`/`ra`/`ndp`=disabled) + удаление ULA + `odhcpd disable/stop`.
- **WAN-детект профиле-зависимый**: модем через **ModemManager** (`mmcli`), eth-WAN
  для дома, fallback по default route; ручной override `wan_iface`.
- HW/SW **flow offload off** (несовместимы с AQM + конфликт с policy-routing Podkop).
- `kmod-tcp-bbr` остаётся зависимостью (стандартное ядро); **`irqbalance` убран из
  зависимостей** (всегда off — на 2 ядрах нейтрально/вредно).
- Кнопка **«Определить оптимальный MTU»** (rpcd `probe_mtu`).
- BBR-версия по размеру `tcp_bbr.ko` (modinfo в OpenWRT вырезан).

## 1.3
- **Определение версии BBR переделано на размер модуля.** OpenWRT вырезает
  `.modinfo` из модулей, поэтому `version`/`srcversion` отсутствуют — прежний
  способ (sysfs/modinfo) на роутере всегда давал «unknown». Теперь версия
  определяется по размеру stripped-модуля `tcp_bbr.ko` (патчи v3 крупнее стока
  v1): порог `bbr_v1_maxsize` (UCI, дефолт 14500 — для ImmortalWRT 25.12
  mediatek, где сток v1 ≈ 13856 Б, v3 ≈ 15736 Б). В UI показывается и сам размер
  как доказательство. На не-OpenWRT ядрах сначала пробуется sysfs `version`.
